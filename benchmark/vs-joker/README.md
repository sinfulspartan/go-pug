# go-pug vs Joker/jade

A standalone, three-way render-throughput comparison between go-pug and
[Joker/jade](https://github.com/Joker/jade) (`github.com/Joker/jade`), a
mature, independently developed Pug/Jade template engine for Go.

**This is a separate cross-library comparison, not part of the byte-identity-gated
3-way corpus in `../README.md`.** It lives in its own Go module
(`benchmark/vs-joker/go.mod`) specifically so the root go-pug module never
gains a dependency on Joker/jade — see [Why a separate module](#why-a-separate-module).

Three engines, same four templates as the main corpus (`card_list`, `table`,
`form`, `blog`):

- **go-pug interpreter** — `gopug.Template.Render`, the default runtime path
- **go-pug codegen** — a generated Go render function (`gopug.GenerateGo`), built once and called directly
- **Joker/jade precompiled** — a generated Go render function (the `jade` CLI's `-writer` output), built once and called directly — Joker's fast path, not its `html/template`-generating runtime API

## The headline, honestly

**go-pug codegen is 2.5–6.8x faster than Joker's compiled path. go-pug's interpreter is 1.2–3.3x *slower* than Joker on all four templates.** That interpreter gap narrowed this release (it was 3–3.6x on three templates and ~1.4x on `blog` previously) thanks to buffer/scope pooling and a mixin-argument fix landed since the last measurement here — but the interpreter is still slower than Joker on every template. The win here is codegen, not the interpreter — do not read this page as "go-pug beats Joker"; read it as "go-pug's ahead-of-time-compiled path beats Joker's ahead-of-time-compiled path, and go-pug's default interpreted path does not."

## Mechanism

Joker's precompiled functions box every interpolated value into `interface{}`
(`WriteAll(a interface{}, escape bool, buffer *WriterAsBuffer)` in
`joker_runtime.go`, generated by the jade CLI) and dispatch on a type switch
at render time, then escape the resulting string byte-by-byte
(`WriteEscString`, scanning for `<>"'&` one rune at a time). Every one of
those `interface{}` conversions for a non-pointer, non-interned value is a
heap allocation; the type switch and per-byte escape loop cost CPU on top of
that.

go-pug's codegen output is monomorphized: it knows every field's static Go
type at generate time; `d.PageTitle` is written as a plain typed function
call (`gopug.EscapeHTML(d.PageTitle)`, a `string`-in-`string`-out scan that
returns the original string unmodified when nothing needs escaping, so the
common case allocates nothing) with no boxing and no runtime type dispatch.
That's the entire difference visible in the allocs/op table: go-pug codegen
sits at 1–6 allocs/op, Joker at 20–62.

## Output fidelity

`card_list`, `table`, and `blog` are byte-identical across all three engines.
`form` differs by exactly 5 bytes: Joker self-closes void elements
(`<input .../>`), go-pug and pug.js render them HTML5-terse (`<input ...>`);
`form` has five `<input>` elements, one byte each. Both are valid HTML5 —
this is a legitimate, cosmetic rendering-convention difference, not a
correctness bug in either engine. `main.go`'s own fidelity check normalizes
this one known difference and still asserts identity on top of that
normalization, so a *new*, unexpected divergence would fail loudly rather
than being silently swallowed alongside the known one.

## Porting to Joker's dialect

Joker/jade shares pug.js's tag/attribute/each/if/case syntax closely enough
that three of the four templates ported with no changes beyond the
`:go:func` typed-parameter preamble Joker's CLI needs (its own dialect
feature, not present in plain Pug/pug.js) and dotted field access
(`d.PageTitle` instead of a bare local). One construct in the source corpus
does not port as-is:

```pug
// ../templates/card_list.pug (go-pug, pug.js) — this exact form is valid pug.js
div(class=(product.Featured ? "product-card product-card--featured" : "product-card"))
```

```jade
// templates_joker/card_list.jade (Joker) — the parenthesized form above
// does not compile under Joker; a bare `cond ? a : b` does, but this project
// wrote it as an explicit call to the `ternary` helper the jade CLI itself
// generates into every package, to make the rewrite unambiguous rather than
// relying on removing exactly one pair of parens
div(class=ternary(product.Featured, "product-card product-card--featured", "product-card"))
```

`table.pug`'s pug.js-specific class-merging object literal has no Joker
equivalent at all and needed the same `ternary` rewrite:

```pug
// ../templates/table.pug (go-pug, pug.js)
tr(class={"data-table__row": true, "data-table__row--highlight": employee.Highlight})
```

```jade
// templates_joker/table.jade (Joker)
tr(class=ternary(employee.Highlight, "data-table__row data-table__row--highlight", "data-table__row"))
```

Both rewrites render identically to the original — see [Output fidelity](#output-fidelity) — so this is a syntax-compatibility gap being worked around, not a semantic difference being measured.

## Disclosed tweak

`joker_runtime.go`'s `WriterAsBuffer.WriteString`/`WriteByte` are one hand-edit off the jade CLI's verbatim generated output: they take the `io.StringWriter`/`io.ByteWriter` fast path when the wrapped `io.Writer` supports it (a `*bytes.Buffer` does), instead of unconditionally allocating a fresh `[]byte` per fragment written (the CLI's stock output does `w.Write([]byte(s))` on every call). This is the **one** disclosed, Joker-favorable change in this entire comparison — it makes Joker faster and lowers its allocs/op, which understates go-pug's lead rather than overstating it. It exists to put both compiled engines on equal footing: go-pug's own generated code already gets this exact optimization for free, because `io.WriteString(w, s)` (what its codegen emits) already checks for `io.StringWriter` internally. Without this tweak, Joker would be penalized by an implementation detail unrelated to its actual template-rendering approach.

## Joker's real strengths

This benchmark measures one thing narrowly — Joker's precompiled-function fast path, rendering four small fixed templates on one machine. It does not capture where Joker is a stronger choice for many projects:

- **More mature and more widely used** than go-pug — a longer track record, more production usage, more of the long tail of Pug/Jade syntax covered.
- Its `html/template` integration path (`jade.Parse` + `template.Execute`, not measured here) gets Go's context-aware HTML escaping — it knows whether a value is being written into an attribute, a `<script>` block, a URL, etc., and escapes accordingly. go-pug's escaping is uniform (HTML-entity escaping in text/attributes, no injected context grammar) and does not yet have that context-sensitivity.
- The precompiled path measured here is Joker's *fastest* mode; its runtime `html/template`-generation mode (not measured) trades some of that speed for the escaping guarantees above and is likely the more common way it's actually used in an HTTP handler.

## Why a separate module

The root go-pug module (`go.mod` at the repository root) declares **zero**
`require`s — see the root README's "No external dependencies" line. Adding
Joker/jade as a dependency anywhere `pkg/gopug` or the root module can see it
would break that promise for every consumer of `github.com/sinfulspartan/go-pug`,
even ones who never look at this comparison. `benchmark/vs-joker/` is its
own Go module, with its own `go.mod`/`go.sum`, that `replace`s
`github.com/sinfulspartan/go-pug => ../..` to build against this checkout. A
nested module is invisible to the parent module's `./...` — running `go
build ./...`, `go vet ./...`, or `go mod tidy` at the repository root never
descends into `benchmark/vs-joker/`, so the root `go.mod`/`go.sum` are
provably unaffected by anything in this directory.

## Fairness caveats

- **Single-machine, cross-library microbenchmark.** Absolute renders/second is specific to this machine (AMD Ryzen 9 5900HX, Windows/amd64, go1.26.0); only the relative ordering and rough magnitude should be expected to reproduce elsewhere.
- **Templates were ported**, not shared source — see [Porting to Joker's dialect](#porting-to-jokers-dialect). The two rewritten constructs render identical output to the originals (confirmed by the fidelity check), so this is a syntax accommodation, not a semantic change to what's being measured.
- **`form` differs by 5 bytes** from the other two engines for a known, disclosed, cosmetic reason (void-element self-closing) — see [Output fidelity](#output-fidelity).
- **This is Joker's fastest path only** — see [Joker's real strengths](#jokers-real-strengths) for what isn't being measured.

## Reproduce

```sh
cd benchmark/vs-joker
go run .
```

Requires Go on `PATH`; the module's own `go.mod`/`go.sum` pin Joker/jade v1.1.3 and its transitive dependencies, resolved independently of the root module.

To regenerate the generated engine code from source:

```sh
# go-pug codegen: regenerates codegen_card_list.go, codegen_table.go, codegen_form.go, codegen_blog.go
# from ../templates/*.pug
go run ./gengopug

# Joker/jade precompiled: regenerates card_list.jade.go, table.jade.go, form.jade.go, blog.jade.go
# (rename to joker_*.go afterward) and jade.go (the runtime helpers — reapply the
# disclosed tweak in "Disclosed tweak" above; joker_runtime.go's own comment
# documents exactly what changed) from templates_joker/*.jade. All four
# templates must be passed to a single invocation — the CLI collapses
# identical string-literal constants (the shared doctype/head/title
# boilerplate) across files generated in the same invocation, so generating
# them one at a time produces undefined constant references between files.
go tool jade -writer -pkg main -d . templates_joker/card_list.jade templates_joker/table.jade templates_joker/form.jade templates_joker/blog.jade
```

To regenerate `chart.svg` from `results.json` (the single source of truth for both the chart and the table above — update `results.json` first, then regenerate, rather than editing the chart or the table by hand):

```sh
go run ./chartgen
```

## Methodology

The timing scheme mirrors `../main.go`'s (calibrate, then warm up and time 5
repetitions, report the median), with one deliberate difference documented
in `main.go`'s own comment: calibration here times a batch that *doubles in
size* until it takes at least 20ms, rather than always timing a fixed 30
iterations. go-pug codegen and Joker's precompiled path both render some of
these templates in a few microseconds, and 30 of those can complete faster
than this machine's timer reliably distinguishes — this was caught during
development of this benchmark, where a fixed-iteration calibration
occasionally measured an exact zero duration and inflated the estimated
render rate by several orders of magnitude, in turn inflating the actual
per-repetition iteration count to its hard cap and turning a
sub-two-second measurement into one that took minutes. Doubling the batch
size until it's reliably measurable avoids that failure mode regardless of
how fast the engine being measured is.

1. **Pre-compile/pre-generate once, outside the timed loop.** The interpreter via `gopug.Compile`; codegen via `gopug.GenerateGo`, generated once ahead of time via `go run ./gengopug` and committed (see `codegen_*.go`); Joker via the jade CLI, generated once ahead of time and committed (see `joker_*.go`).
2. **Time only the render call**, into the same reused `*bytes.Buffer` for codegen and Joker (`Reset()` between calls); the interpreter's API returns a string per call (it has no writer-based render path), so its allocation includes that string.
3. **Calibrate with a duration-based batch** (see above), then measure 5 repetitions, each preceded by its own discarded warmup (15% of the sized iteration count), and report the median renders/second and median ns/op.
4. **Allocations per render** are sampled separately via `testing.AllocsPerRun` (a public API usable outside `go test`), which disables the GC around a batch of calls and reports the average.
5. **Fidelity checked before trusting any timing number** — see [Output fidelity](#output-fidelity).
